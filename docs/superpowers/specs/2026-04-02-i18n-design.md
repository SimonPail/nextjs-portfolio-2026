# i18n Design — EN/FR Internationalization

## Overview

Internationalize the portfolio with English and French using `next-intl`. Locale is stored in a cookie (no locale prefix in URLs). Browser language is detected on first visit, defaulting to English.

## Technical Approach

**Library:** `next-intl` with `localePrefix: 'never'`  
**Locales:** `en` (default), `fr`  
**Locale storage:** `NEXT_LOCALE` cookie (set by middleware on first visit, updated by locale switcher)  
**Detection:** Middleware reads `Accept-Language` header on first visit, falls back to `en`

## Configuration Files

### next.config.ts
Wrap existing config with `createNextIntlPlugin` from `next-intl/plugin`.

### i18n/routing.ts
```ts
defineRouting({
  locales: ['en', 'fr'],
  defaultLocale: 'en',
  localePrefix: 'never'
})
```

### i18n/request.ts
- Reads locale from cookie
- Validates against supported locales using `hasLocale()`
- Falls back to `en` if invalid/missing
- Dynamically imports `messages/{locale}.json`

### middleware.ts
- Uses `createMiddleware(routing)`
- Matcher excludes `api`, `_next`, `_vercel`, static files
- Internally rewrites URLs to include `[locale]` segment (invisible to user)

## Locale Switcher

- Nav EN/FR buttons call a server action that sets the `NEXT_LOCALE` cookie
- After setting, `router.refresh()` re-renders with new locale
- Active locale button is visually highlighted (bold + dark text)
- PT button removed from mobile menu (EN/FR only)

## Contact Form Validation

- Zod schema becomes a factory: `createContactSchema(messages)` accepting translated error strings
- Server action receives locale via hidden `<input name="locale">` field
- Loads translated messages server-side for that locale
- Returns translated validation errors and success/error messages

## What Stays Untranslated

- Proper nouns: "Simon Paillassa", company names ("ITAF.TECH", "Addeo", etc.)
- Tech stack tags: "React.js", "Next.js", "TypeScript", etc.
- URLs and email addresses
- External link labels: "GitHub", "LinkedIn"
- Email template HTML sent via Resend (only seen by Simon)

## SEO

- `<html lang={locale}>` set dynamically in layout
- No `hreflang` alternate links (same URLs for both locales — acceptable for a portfolio)
- Page metadata translated via `generateMetadata` using `getTranslations()`

## Edge Cases

- Invalid/missing cookie: falls back to English
- Unsupported locale value: validated with `hasLocale()`, falls back to English
- First visit: middleware detects `Accept-Language`, sets cookie, serves matching locale
